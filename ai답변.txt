 멘토 피드백 기준으로 추가 진행할 일은 아래처럼 정리하면 됩니다.

  1. 적치 추천 정규화 고도화

  현재 적치 점수는 same_sku_score + capacity_score + distance_score + fast_moving_score - congestion_penalty 구조라서, 특정 항목이 과도하게 점수를 지배할 수 있습니다. 추가로 해야 할 일은 다음입니다.

  - 보관조건, CAPA 부족, 사용불가 Location은 계속 hard filter로 유지
  - 점수 항목은 모두 0~1 또는 0~100으로 정규화
  - 거리, 잔여용량, 혼잡도는 창고/Zone별 기준값으로 min-max 또는 clipped normalization 적용
  - 동일 SKU는 절대 우선인지, 가중치 우선인지 정책 결정
  - 최종 산식 예:

    score =
      0.30 * same_sku_norm
    + 0.25 * capacity_norm
    + 0.20 * distance_norm
    + 0.15 * turnover_norm
    - 0.10 * congestion_norm

  - 추천 결과에는 총점뿐 아니라 항목별 breakdown을 제공
  - 가중치 변경 시 추천 순위가 어떻게 바뀌는지 sensitivity test 추가

  문서 반영 위치: rag/scoring_formula.md, docs/06_TOOL_DESIGN.md, docs/10_EVALUATION_PLAN.md

  2. DES 활용 컨셉 명확화

  현재 문서의 재고 리스크 예측은 단순 수량 기반 Forecast에 가깝습니다. SimPy DES를 넣으려면 목적식과 제약사항을 추가해야 의미가 생깁니다.

  DES의 목적은 “수요 예측” 자체가 아니라, 예측된 입출고 이벤트가 실제 창고 운영 제약 안에서 처리 가능한지 검증하는 것입니다.

  추가해야 할 목적식:

  - 재고 소진 시점 예측
  - 출고 지연 발생 여부
  - 피킹/패킹/출고 대기 Queue 최소화
  - Zone 포화/적치 실패 위험 탐지
  - 작업자/지게차/피킹 리소스 사용률 분석
  - What-if 조건별 서비스레벨 비교

  추가해야 할 제약사항:

  - 작업자 수
  - 지게차/장비 수
  - 입고 처리시간
  - 적치 처리시간
  - 피킹 처리시간
  - 패킹/출고 처리시간
  - Zone별 CAPA
  - 출고 마감시간
  - 입고 예정 지연 가능성
  - 피킹 우선순위 정책

  추가기획.txt의 Hybrid Forecast 방향은 적절합니다. 정리하면:

  Near Future:
  확정 입고/출고 + 현재 재고 + 운영 리소스 제약 → SimPy DES

  Far Future:
  Regression 수요예측 → 가상 출고 이벤트 생성 → SimPy DES

  즉, 단기는 확정 이벤트 기반 시뮬레이션, 장기는 Forecast 결과를 이벤트로 변환해 DES에 넣는 구조가 좋습니다.

  추가 Tool 후보:

  - run_des_simulation
  - simulate_inventory_risk_des
  - simulate_operation_what_if
  - compare_simulation_scenarios

  출력 KPI:

  - 예상소진일
  - 안전재고 도달일
  - 출고 지연 주문 수
  - 평균 피킹 대기시간
  - Zone별 최대 점유율
  - 작업자/장비 사용률
  - 병목 이벤트 타임라인

  문서 반영 위치: docs/07_FORECAST_AND_SIMULATION.md, docs/06_TOOL_DESIGN.md, docs/04_DATABASE_DESIGN.md, docs/09_UI_DESIGN.md

  3. What-if 시뮬레이션 고도화

  현재 What-if는 “입고 추가”와 “수요 증가” 정도입니다. DES를 넣으면 운영 조건 변경까지 비교할 수 있어야 합니다.

  추가 What-if 예시:

  - 작업자 1명 추가 시 출고 지연이 줄어드는가
  - 지게차 1대 감소 시 적치 대기가 얼마나 증가하는가
  - 특정 Zone CAPA가 20% 줄면 병목이 생기는가
  - A제품 출고량이 30% 증가하면 언제 부족해지는가
  - 입고가 하루 지연되면 출고 실패가 발생하는가
  - 피킹 우선순위를 변경하면 긴급 주문 지연이 줄어드는가

  결과는 baseline vs scenario 비교로 제공하면 됩니다.

  Baseline:
  - 예상소진일: 2026-06-18
  - 출고지연: 3건
  - 피킹 대기 평균: 42분

  Scenario:
  - 예상소진일: 2026-06-21
  - 출고지연: 1건
  - 피킹 대기 평균: 18분

  4. RAG 품질 및 Reasoning 보강

  현재 RAG는 vector search + optional rerank 수준입니다. 멘토 피드백을 반영하려면 “검색 결과가 충분한가”를 판단하는 단계가 필요합니다.

  추가할 구조:

  Query
  → Intent/required evidence 정의
  → 1차 검색
  → Rerank
  → Sufficient Context Judge
     ├─ 충분함 → 답변 생성
     └─ 부족함 → query rewrite / 추가 검색 / 문서군 확장
  → Evidence-grounded response

  참고할 개념:

  - Google 계열 연구의 Sufficient Context는 검색된 context가 질문에 답하기 충분한지 판단하고, 부족하면 생성 대신 abstain 또는 재검색하는 방향입니다.
    Source: https://arxiv.org/abs/2411.06037

  - AgenticRAG는 단발 검색이 아니라 search/find/open/summarize 같은 도구를 반복 사용해 evidence를 찾는 harness 구조를 제안합니다.
    Source: https://arxiv.org/abs/2605.05538

  - PRISM Reranker는 단순 relevance score뿐 아니라 contribution과 evidence passage를 함께 산출하는 방향이라, 추천 근거 설명에 잘 맞습니다.
    Source: https://arxiv.org/abs/2604.23734

  추가 구현 항목:

  - context_sufficiency_score
  - missing_evidence_types
  - required_sources
  - evidence_span
  - reranker_contribution
  - answerable: true/false
  - 근거 부족 시 “문서 근거가 부족합니다” 응답 정책

  DOCSEEKER는 정확히 어떤 모델/도구를 지칭하는지 명칭 확인이 필요합니다. 현재 우선순위는 PRISM 또는 일반 cross-encoder reranker 검토가 더 명확합니다.

  문서 반영 위치: docs/03_RAG_DESIGN.md, docs/02_AGENT_ARCHITECTURE.md, docs/10_EVALUATION_PLAN.md

  5. Harness Engineering 추가

  평가 문서에 단순 정확도뿐 아니라 반복 실험 가능한 harness를 추가해야 합니다.

  추가할 harness:

  - Tool execution harness: 같은 입력이면 같은 Tool 결과가 나오는지 검증
  - RAG harness: 질문별 expected source/evidence span 검증
  - DES simulation harness: seed 고정 후 KPI 재현성 검증
  - What-if harness: baseline/scenario delta 검증
  - Answer grounding harness: 최종 답변의 수치가 Tool 결과와 일치하는지 검증
  - Regression test set: 멘토 피드백 반영 후 기존 기능이 깨지지 않는지 확인

  6. Dreaming Memory는 차순위로 설계만 반영

  현재 문서의 메모리는 최근 10턴 + 구조화 슬롯입니다. POC에서는 이대로 가고, Dreaming Memory는 차순위 Enhancement로 두는 게 맞습니다.

  추가 설계만 해둘 내용:

  - 세션 종료 후 비동기 memory consolidation 수행
  - 대화 전체를 그대로 저장하지 않고, 안정적인 운영 선호/반복 이슈/자주 언급 SKU만 요약
  - 예: “사용자는 출고 임박 건을 재고 리스크보다 우선 확인하는 경향”
  - 메모리 저장 전 필터링: 개인정보, 일회성 지시, 잘못된 추론 제외
  - 장기 메모리는 답변 근거가 아니라 개인화/업무 편의용으로만 사용

  문서 반영 위치: docs/02_AGENT_ARCHITECTURE.md, docs/12_IMPLEMENTATION_ROADMAP.md

  우선순위 제안

  1. 적치 점수 정규화 산식 확정
  2. DES 목적식/제약사항/출력 KPI 정의
  3. Hybrid Forecast + SimPy 구조를 07_FORECAST_AND_SIMULATION.md에 반영
  4. What-if 시나리오 범위 확장
  5. RAG에 Sufficient Context Judge와 reranker 검토 추가
  6. Evaluation Harness 설계 추가
  7. Dreaming Memory는 Phase 10 이후 차순위로 명시

  핵심은 DES를 “재고 수량 예측 도구”로 포장하지 않는 것입니다. DES는 운영 제약을 반영해 “예측된 수요와 확정 주문을 실제 창고가 처리 가능한가”를 검증하는 시뮬레이션 엔진으로 정의하는 편이 가장 자연스럽습니다.
